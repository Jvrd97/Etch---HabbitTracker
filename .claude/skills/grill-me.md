# Grill Me — Alignment Ritual

## Цель

Добиться **shared understanding** между пользователем и агентом *до* того, как написана хоть одна строка кода или сделан план. Misalignment в начале = мусор на выходе.

## Процесс

0. **Язык грилла — русский.** Весь диалог (вопросы, рекомендации, реплики) ведём на русском, см. CLAUDE.md §8. Английскими остаются только идентификаторы, имена сервисов/таблиц, код и conventional-commit префиксы.

1. **Если есть кодбаза** — запусти sub-agent на explore (ОДИН раз, в начале). Не читай весь репо в основной контекст.

2. **Сформулируй design tree** мысленно: какие основные ветки решений нужно принять (data model, UX flow, edge cases, scale, security, ops).

3. **Иди по дереву сверху вниз**. По одному вопросу за раз. **Каждый вопрос идёт со своей рекомендацией.**

   Формат:
   ```
   Q: <конкретный вопрос с вариантами>
   Recommendation: <твоя рекомендация и почему>
   ```

4. **Не объединяй вопросы**. Один вопрос = одно решение. Это снижает когнитивную нагрузку и упрощает откат.

   **Вопрос держи коротким (по умолчанию `/caveman lite`).** Структура: вопрос + варианты + рекомендация. Без преамбул и пересказа кодбазы. Цель грилла — alignment, а не лекция; длинные простыни замедляют ответ и топят суть.

5. **Не пиши план**. Пользователь сам скажет, когда хватит. Если он молчит — продолжай вопросы.

6. **Прекращаешь** когда:
   - Пользователь явно говорит "достаточно" / "пиши PRD" / "поехали"
   - Все ветки design tree закрыты (вопросов больше нет)

## Lifecycle айтема

Если грилл стартует на айтеме, который **лежит в папке `backlog/`** (PRD / ADR / concern / triage-item) — в самом начале сессии `mv` его в `in-work/` того же типа. Это «я взял это в работу».

По завершении грилла: если из айтема родился результат — `mv` в `done/`. Если по ходу выяснилось, что айтем не тот / не нужен — проставь `status: rejected`, допиши секцию `## Rejected (YYYY-MM-DD)` с причиной и `mv` в `rejected/`.

`issues/` в `.gitignore` — переносы это обычный `mv`, не коммитим. Подробности — CLAUDE.md §1.

## Категории вопросов (минимум по одному из каждой)

- **Data model**: какие сущности, ключи, связи, индексы
- **State management**: что в БД, что в кэше, что в контексте, что в файле
- **User flow**: точная последовательность шагов с edge cases
- **Failure modes**: что если запрос не дошёл, БД упала, LLM вернул мусор
- **Scale**: ожидаемый QPS, размер данных, рост за год
- **Out of scope**: что мы явно НЕ делаем в этом PRD
- **Testing**: где границы тестов, какие fixtures, какой test DB

## Запрещено

- ❌ Сразу выдавать план без вопросов
- ❌ Один большой вопрос с 10 подпунктами
- ❌ Вопросы без своей рекомендации (это перекладывание мышления на пользователя)
- ❌ Уточнения по реализации (это для имплементации, не для alignment)

## Хороший пример вопроса

```
Q: Начислять ли очки геймификации задним числом за уже пройденные уроки?
   У нас есть completed_at по всем прошлым прохождениям.

Рекомендация: Да, backfill при первом логине каждого юзера (не bulk-миграцией).
Почему: те, кто уже прошёл уроки, сразу чувствуют награду; backfill на логине
размазывает нагрузку и убирает одну большую миграцию. Кап — последние 30 дней,
чтобы не отсыпать тысячи очков старым юзерам.
```

## Плохой пример

```
What do you want? Tell me about the requirements, scale, edge cases,
data model, and testing approach.
```
(Слишком общё, нет рекомендаций, перекладывает работу на пользователя.)
