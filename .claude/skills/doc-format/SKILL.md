# Long-Form Doc Format (ADR / PRD / README / issues)

Эти документы читает уставший человек, который ищет «что решили и что делать».
Оптимизируй под сканирование. Структуру секций задают adr-format / шаблоны PRD —
этот скилл про то, как выглядит текст внутри секций.

## Hard rules

1. **Абзац ≤ 3 предложений.** Четвёртое предложение — сигнал: разбей на bullets
   или новый абзац.
2. **Один факт = один bullet.** Не склеивай факты через `;` внутри bullet'а или
   абзаца. Перечисление из 3+ элементов в прозе → список.
3. **Решение из N частей = N подзаголовков `####`** (или нумерованных
   block-абзацев ≤ 3 предложений каждый), не один мега-абзац на часть.
4. **Table cells ≤ 8 слов.** Предложения — в bullets под таблицей, не в ячейках.
5. **AC: один критерий = один checkbox.** Если внутри `- [ ]` есть `;` с двумя
   проверяемыми фактами — раздели на два checkbox.
6. **Пустая строка между блоками** (заголовок / список / абзац / таблица).
7. **Bold — только смысловое выделение**: вердикты, severity, имена файлов,
   которые нельзя пропустить. Не для каждого термина.
8. **Скобки-вставки ≤ 1 на предложение.** Вторая вложенная скобка — переписать.
9. **Ссылки по месту**, а не «см. выше/ниже»; id документов текстом + relative
   link внутри repo.

## Шаблон «решение в N частей» (ADR Decision, PRD Solution)

```markdown
#### 1. <Короткое имя части>

<1-3 предложения сути.>

- <деталь / следствие>
- <деталь / следствие>
```

## Forbidden patterns

- ❌ Абзац на 6+ строк без единого переноса смысла
- ❌ `**Label:** <длинный абзац>` вместо подзаголовка + bullets
- ❌ Checkbox с 3+ фактами, склеенными через `;` или скобки
- ❌ Перечисление 5+ файлов/модулей в прозе (это список или таблица)
- ❌ Ячейка таблицы с двумя предложениями
