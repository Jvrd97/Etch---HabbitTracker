## Knowledge Lecture Generator

### Цель

Превратить разрозненный диалог (вопросы, ответы, дебаг, рефакторинг) в единый структурированный учебный документ, который можно перечитывать как самостоятельную лекцию.
Когда использовать

Пользователь просит собрать пройденное в лекцию / конспект / шпаргалку
Пользователь хочет зафиксировать знания из текущего или прошлых разговоров
Пользователь просит обучающий документ по теме, которую разбирали

Формат выходного документа
Создавай Markdown-файл (.md), который следует структуре ниже. Сохраняй в /mnt/user-data/outputs/.
Обязательная структура
# Тема: [Название]

## Часть N — [Название блока]

### Концепция
Объяснение "что это" и "зачем" — 2-4 абзаца простым языком.

### Как это работает
Механика: что происходит под капотом, в каком порядке.

### Код
Рабочие примеры. Сначала ❌ неправильный вариант (если был в разговоре), потом ✅ правильный.

### Подводные камни
Конкретные ошибки и баги, которые всплыли в разговоре.

## Чеклист
- [ ] Пункт 1
- [ ] Пункт 2
Правила контента

Всё из разговора — не выдумывай примеры, бери реальный код и ситуации из диалога
Контекст ошибок — если в разговоре был баг или непонимание, опиши его: что было неправильно → почему → как исправили
Два уровня — каждая тема раскрывается на двух уровнях:

Общая идея (для быстрого перечитывания)
Точечный разбор (конкретный код, конкретная ошибка)


Поток данных — если тема связана с архитектурой, покажи путь данных (ASCII-диаграмма или текстовая схема)
Связи между темами — если часть A влияет на часть B, укажи это явно
Язык — пиши на том же языке, на котором шёл разговор

Правила форматирования

Код всегда в блоках с указанием языка (python, bash и т.д.)
❌ / ✅ для сравнения неправильного и правильного подхода
Таблицы — только когда нужно сравнить 2+ варианта по параметрам
ASCII-схемы для потоков данных и архитектуры
Чеклист в конце — краткие actionable пункты

Чего НЕ делать

Не превращай в справочник — это лекция, а не документация
Не добавляй информацию, которой не было в разговоре (если только она не критична для понимания контекста)
Не дублируй одно и то же разными словами
Не пиши "как мы обсуждали ранее" — документ должен читаться самостоятельно
Не используй заголовки глубже H3 (###)

Процесс создания

Прочитай весь разговор / указанные пользователем части
Выдели ключевые темы и их логический порядок
Для каждой темы собери: концепцию, примеры кода, ошибки, решения
Расположи темы от базовых к продвинутым
Добавь связи между темами
Собери чеклист из практических выводов
Создай .md файл и покажи через present_files

Пример минимальной структуры
markdown# FastAPI: Зависимости и JWT

## Часть 1 — Depends: три уровня

### Концепция
Depends — механизм DI в FastAPI...

### Как это работает
FastAPI рекурсивно обходит цепочку зависимостей...

### Код

❌ Передаём фабрику вместо готовой функции:
\`\`\`python
router = APIRouter(dependencies=[Depends(get_current_user)])  # видит сигнатуру фабрики
\`\`\`

✅ Передаём результат вызова фабрики:
\`\`\`python
require_user = get_current_user(settings, service_name)
router = APIRouter(dependencies=[Depends(require_user)])
\`\`\`

### Подводные камни
- Swagger не показывает замочек если FastAPI не видит HTTPBearer в сигнатуре

## Чеклист
- [ ] Depends(...) — передавай конечную функцию, не фабрику
